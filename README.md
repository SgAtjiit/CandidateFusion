# Multi-Source Candidate Data Transformer (CandidateFusion)

A robust, 100% deterministic, and explainable data-transformer engine built for the **Eightfold Engineering Intern (Jul-Dec 2026) Assignment**. It ingests semi-structured ATS JSON payloads and unstructured resumes (PDF/DOCX), normalizes and merges them into one canonical candidate profile, and projects the profile using a runtime dynamic schema projection layer.




## 🚀 How to Run

### 1. Prerequisites
- **Node.js**: v18.0.0 or higher is recommended.
- **npm**: Installed automatically with Node.js.

### 2. Installation
Clone the repository and install dependencies:
```bash
npm install
```

### 3. Run the Test Suite
Run the test runner to process all sample inputs (JSON payloads and optional matching PDF resumes) from `sample_inputs/`, apply their schema projections from `sample_configs/`, and write the final unified candidate profiles to `sample_outputs/`:
```bash
npm test
```

### 4. Boot the Web Dashboard
Start the Express server locally:
```bash
npm start
```
Once booted, open your browser and navigate to:
👉 **[http://localhost:3000](http://localhost:3000)**

## 📐 Technical Architecture & Detailed Specification

### 1. Ingestion & Unification Pipeline
CandidateFusion runs a 10-stage sequential transformation pipeline where each stage is pure, deterministic, and independently testable:
1. **INPUT**: Ingests semi-structured ATS JSON payloads or unstructured Resume files (PDF/DOCX).
2. **DETECT SOURCE**: Dynamically routes to the correct parser module based on file headers and extensions.
3. **EXTRACT**: Parses and extracts raw names, contact arrays, links, experience blocks, education, and skills.
4. **NORMALIZE**: Standardizes strings, dates, locations, and skills.
5. **RESOLVE IDENTITY**: Clusters records using a deterministic cascading matching priority.
6. **MERGE**: Merges candidate attributes based on defined field-level master source strategies.
7. **ASSIGN CONFIDENCE**: Evaluates attribute source match conditions and flags consensus agreement.
8. **PROJECTION ENGINE**: Applies schema layout configurations dynamically at runtime.
9. **SCHEMA VALIDATION**: Ensures expected type checks and required fields.
10. **OUTPUT**: Emits the canonical candidate JSON rich with confidence scores and provenance history.

#### Design Principles
- **Pure deterministic pipeline**: Each stage is completely isolated and independently testable.
- **Immutable canonical record**: Once merged, candidate profile values are immutable.
- **Reshaping isolation**: The projection layer only reshapes and reformats output values without editing candidates.
- **Fault-tolerance**: Missing, partial, or corrupted source documents never halt pipeline processing.

---

### 2. Canonical Candidate Schema
All candidate profiles are structured in an immutable canonical structure before projection:

```typescript
interface CanonicalCandidate {
  candidate_id: string;
  full_name: string | null;
  emails: string[];
  phones: string[];
  location: {
    city: string | null;
    region: string | null;
    country: string | null;
  };
  links: {
    linkedin: string | null;
    github: string | null;
    portfolio: string | null;
    other: string[];
  };
  headline: string | null;
  years_experience: number | null;
  skills: Array<{
    name: string;
    confidence: number;
    sources: string[];
  }>;
  experience: Array<{
    company: string | null;
    title: string | null;
    start: string | null;
    end: string | null;
    summary: string | null;
  }>;
  education: Array<{
    institution: string | null;
    degree: string | null;
    field: string | null;
    end_year: number | null;
  }>;
  provenance: Array<{
    field: string;
    source: string;
    method: string;
  }>;
  overall_confidence: number;
}
```

---

### 3. Normalization Standards
Each attribute runs through strict clean-and-parse guidelines to ensure zero data pollution:
- **Emails**: Lowercased and deduplicated.
- **Phones**: Normalized to standard **E.164** format (e.g. `+15550199`).
- **Dates**: Normalized to standard `YYYY-MM` format (unknown months are never invented).
- **Countries**: Mapped to standard **ISO-3166 Alpha-2** codes.
- **Skills**: Mapped to canonical skills list using static synonym dictionary overrides.
- **Unknown/Null Values**: Preserved strictly as `null` or empty arrays `[]` rather than guessing or fabricating dummy values.

---

### 4. Identity Resolution, Merge & Confidence

#### Identity Resolution Priority Order
When linking incoming profiles to existing candidates, CandidateFusion follows a strict cascading precedence:
1. **Candidate ID Match** (Direct ID string matches or matched resume filenames).
2. **Email Overlap** (Intersection check across email arrays).
3. **Phone Overlap** (Intersection check across normalized phone arrays).
4. **Exact Full Name Match** (Case-insensitive alphanumeric name match).
*If all checks yield zero matches, a new candidate profile is created.*

#### Merge Strategy Table
If an ATS record and a Resume match the same candidate, fields are unified based on preferred sources:

| Candidate Attribute | Preferred Source | Strategy Details |
| :--- | :---: | :--- |
| **Name, Email, Phone** | **ATS** | Contact details are fetched from ATS; resume values act as secondary aliases. |
| **Work Experience** | **Resume** | Work history items are unioned and deduplicated. |
| **Education** | **Resume** | Education credentials are unioned and deduplicated. |
| **Skills** | **Union** | Skills from both sources are normalized and deduplicated. |
| **Links** | **Union** | URLs are parsed and unioned across standard structures. |

*Conflicting values are always preserved in the provenance record.*

#### Confidence Scoring Model
Individual fields and records are assigned a confidence score between `0.0` and `1.0`:

| Ingestion Condition | Confidence Score | Details |
| :--- | :---: | :--- |
| **ATS + Resume Agree** | `0.95` | High confidence (complete consensus agreement). |
| **ATS Only** | `0.85` | Good confidence (standard database reference). |
| **Resume Only** | `0.60` | Moderate confidence (heuristic parse reference). |
| **Conflict Resolved** | `0.70` | Lower confidence (due to conflict override). |

* **Overall Confidence**: Calculated as the mathematical average of all populated field confidence scores.

---

### 5. Runtime Projection Engine Capabilities
The projection engine operates dynamically at runtime. It takes the canonical candidates and applies a config supporting:
- **Field Selection**: Output only the requested fields.
- **Field Renaming**: Dynamic renaming via properties mapping (`from` $\rightarrow$ `path`).
- **Runtime Normalization**: Support for inline normalizations (e.g., E164 phone formats or skill synonym lookups).
- **Confidence & Provenance Toggles**: Flexible metadata attachment or omission configurations (`include_confidence`).
- **Missing-Value Policies**: Behavior control via `on_missing` policies (`null`, `omit`, or `error`).
- **JSON Schema Validation**: Strict type-checking (`string`, `string[]`, `number`, `object`, `object[]`) against projected formats.

---

### 6. Robustness & Design Constraints Met
- **Deterministic & Explainable**: Same input always produces identical output, with every value carrying its source and transformation history.
- **Robust**: Missing or noisy inputs never crash the pipeline; unknown information is preserved as null and never fabricated.
- **Scalable**: Linear-time processing per candidate with independent stages enabling parallel ingestion.

## ⚙️ Core Design Choices & Descoped Items

### Core Design Choices
- **Traceability (Provenance)**: Every value is traced back to a specific file and matching method. If two sources disagree, the merge engine logs the conflict override event and saves the secondary source's value, keeping the audit trail transparent.
- **Robustness & Safety**:
  - **Multer Size Limit Protection**: Added middleware catching file uploads larger than 20MB, returning a clean `400 Bad Request` JSON error instead of stack traces or process crashes.
  - **Fail-Safe Defaults ("Honestly empty over wrong-but-confident")**: If resume layout text or date patterns fail regex parses, values are set to `null` or arrays remain empty rather than guessing dates.
  - **Empty Array Resolvers**: Output projection validates index paths (e.g., `emails[0]`) safely against empty arrays `[]` without throwing native JavaScript `TypeError` crashes.

### Descoped Approaches (Trade-offs & Rationale)
- **Sentence Embeddings for Skill Matching**: We deliberately descoped local sentence embeddings (like `all-MiniLM-L6-v2`) in favor of a static synonym dictionary (`js` $\rightarrow$ `JavaScript`) and direct case-insensitive listings. This removes the 90MB weight package download, avoids CPU-bound local model inference latency on startup, and provides 100% auditable, binary-explainable canonical matches.
- **LLM-Based Resume Parsing**: We descoped the use of external LLM APIs (e.g. GPT-4) or heavy NLP frameworks (like spaCy) for parsing resume text. Instead, we use structured regex segmentation and heuristic keyword-matching (categorizing lines under experience, education, and skills sections). This guarantees zero external API dependency, zero run cost, and near-instant processing, avoiding LLM non-determinism.
- **Third-Party Geocoding API**: We descoped third-party geocoding services (such as Google Maps or GeoNames APIs) in favor of a static, localized lookup map of major tech hubs and ISO country codes. This eliminates network round-trips, avoids API key requirements, and ensures offline execution safety.
- **Graph-Based/Fuzzy Identity Resolution**: We descoped complex graph-clustering or probabilistic record-linkage models for matching identities. Instead, we use a deterministic cascading resolution pipeline (matching sequentially on Candidate ID, email array intersection, phone array intersection, and exact name match). This provides completely predictable candidate clusters and makes false-merge bugs trivial to audit and resolve.
